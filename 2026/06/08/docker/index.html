docker

前言

第一次接触docker，也是在CTF中，题目的附件中给出了Dockerfile(or docker-compose.yml)，一开始并不知道有什么用，后面才知道本地调试时要docker build -t xxx .构建一个镜像，再docker run -it xxx启动一个容器，以保证本地与远程的环境一致

故以此为引，开始docker的学习

由来

不论是开发还是CTF，都会遇到一个问题，你本地能运行的代码，别人那里不一定可以，你本地能打通的exp，别人也许一运行就EOF了，要么依赖版本不同，要么系统库存在差异，总之，在部署和运行上，环境不一致带来了很大的麻烦

一开始的解决方案是虚拟机(VM)，但是每个VM都要装一个完整的操作系统，几个GB起步，启动慢，占资源，迁移也麻烦，为了隔离一个可能只有几十MB甚至KB的小应用，硬要扛着几GB的系统内核，怎么看都不划算

于是，docker诞生了，其不模拟硬件，也不装完整系统，而是直接用宿主机的内核，只把应用 + 依赖打包成一个镜像，然后跑起来就是一个容器

概念

那么，什么是镜像，什么是容器呢？

镜像

镜像相当于一个只读的模板，它把代码，运行环境，依赖库，配置文件等等全部打包在一起，你可以编写Dockerfile以拉取或创建一个特定的镜像

Dockerfile背后的层缓存是什么？

构建镜像时，docker会复用之前已经构建过的，未发生变化的层，而不是重新执行该层的指令

容器

容器就是镜像的运行实例，当执行docker run时，镜像就变成了一个活着的容器

容器启动时，docker会在镜像的只读层上面叠一层可写层，你在容器里做的所有修改(写文件，装软件，改配置啊等等)都落在这层上，容器一删，这层可写层也就没了，但原始镜像依然完好，这就是写时复制技术(Copy-on-Write)

关键：容器本质上是一个被限制住的进程，docker并没有启动一个新内核，它只是让一个进程看起来像是运行在一个独立的系统里，你在宿主机上执行ps aux依然能看到这个进程，只不过它被包了一层壳而已

额那这层壳是什么呢

1.Namespace

Namespace让进程只能看到自己的那一份系统资源视图，实现了隔离：

  • PID namespace：容器内的进程以为自己的PID是$1$，看不到宿主机上的其它进程

  • NET namespace：容器有自己独立的网络栈(网卡，路由表，iptables规则)

  • MNT namespace：容器有自己独立的文件系统挂载点

  • UTS namespace：容器可以有独立的主机名和域名

  • IPC namespace：容器有自己的进程间通信资源(共享内存，消息队列)

  • USER namespace：容器内可以有自己独立的用户ID映射

2.Cgroups

Cgroups(Control Groups)负责给进程划定资源上限：

  • CPU配额(比如最多用$2$个核心)

  • 内存限制(超过则触发OOM Killer)

  • 磁盘I/O读写速率限制

  • 网络带宽限制

  • 进程数量限制

3.rootfs

通过chroot，容器进程的/不再是宿主机的/，而是镜像里打包好的那一套文件系统(如Ubuntu，Alpine的根文件系统)，于是，容器内执行/bin/bash用的是镜像里的bash，而不是宿主机上的bash，容器内看到的/etc，/var也都是与宿主机隔离的

容器启动时，没有内核引导，没有init系统，也没有硬件探测，本质上只是fork一个进程，然后在exec之前设置好Namespace，Cgroups，rootfs，所以容器启动是毫秒级的

且容器里最终运行的还是一个普通的进程，即一个应用，一个服务，没有这个进程，容器就会立刻退出

安全

沙箱(sandbox)

沙箱(sandbox)是一种安全隔离机制，它的目标是：让一个进程只能访问被允许的资源，其余的一切都接触不到，哪怕这个进程被攻击者控制了，也无法逃逸到宿主系统

在docker中沙箱由几种机制组合而成，首先便是上述的Namespace与Cgroups，其次还有：

Capabilities

容器里的root不是拥有完整宿主机权限的root，docker默认只保留一小组必要capabilities，并丢弃CAP_SYS_ADMIN，CAP_SYS_MODULE等高危能力

Seccomp

docker默认seccomp profile是syscall allowlist，大约拦截$40$多个危险或不必要的系统调用，例如模块加载，重启，部分namespace操作等

逃逸

虚拟机的沙箱依赖硬件辅助虚拟化(如Intel VT-x)和独立的Guest内核，隔离强度高，但开销大

而docker的沙箱则是内核层面的软件隔离，更轻量，但容器本质上还是共享着宿主机的内核，一旦内核本身存在漏洞，攻击者就有可能逃逸出沙箱，影响宿主机…
